## MVC 패턴이란?

**MVC**는 **Model-View-Controller**의 약자예요.
1979년에 만들어진 오래된 패턴이지만, 아직도 널리 쓰입니다.

레스토랑에 비유해 볼까요?

```
┌──────────────────────────────────────────────────────────┐
│                     MVC = 레스토랑                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│   Model (주방 창고)                                       │
│   └── 재료가 얼마나 있는지 알고 있어요                      │
│   └── "오늘 토마토 30개, 양파 20개 남았어"                  │
│                                                          │
│   Controller (셰프)                                       │
│   └── 주문을 받아서 요리를 만들어요                         │
│   └── "스파게티 주문! 토마토 3개 쓸게!"                     │
│                                                          │
│   View (서빙 직원)                                        │
│   └── 손님에게 음식과 정보를 전달해요                       │
│   └── "손님, 스파게티 나왔습니다~"                         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 각 역할의 책임

### Model (StationData)
```
✅ 할 일:
- 데이터 저장 (비누 잔량, 사용 횟수)
- 데이터 변경 시 알림 발송

❌ 하면 안 되는 일:
- UI 직접 조작
- 버튼 클릭 처리
```

### Controller (StationController)
```
✅ 할 일:
- 비즈니스 로직 처리 (비누 분사 시작/종료)
- 인터락 체크 (동시 작동 방지)
- 파티클 효과 재생

❌ 하면 안 되는 일:
- UI 요소 직접 변경
- 데이터 저장 방법 결정
```

### View (HMIUIController)
```
✅ 할 일:
- 화면에 정보 표시
- 사용자 입력 받기 (버튼 클릭)
- 이벤트를 Controller에 전달

❌ 하면 안 되는 일:
- 비즈니스 로직 직접 구현
- 데이터 직접 수정
```

## 이벤트 기반 설계

### 폴링 방식 vs 이벤트 방식

```
폴링 방식 (비효율적) ❌
─────────────────────────────────────────
매 프레임: "데이터 바뀌었어?"
응답: "아니"
매 프레임: "데이터 바뀌었어?"
응답: "아니"
... (초당 60번 반복!)
매 프레임: "데이터 바뀌었어?"
응답: "응, 바뀌었어!"
→ 드디어 업데이트

이벤트 방식 (효율적) ✅
─────────────────────────────────────────
데이터 변경 발생!
Model: "야, 나 바뀌었어!" (이벤트 발송)
View: "알겠어, 업데이트할게!" (구독자 반응)
→ 바뀔 때만 업데이트
```

## 데이터 흐름 시각화

```
사용자 버튼 클릭
      │
      ▼
┌─────────────────────────────────┐
│     View (HMIUIController)       │
│     "버튼 클릭됐어!"             │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│    Controller (StationController)│
│    "비누 분사 시작하자!"          │
└──────────────┬──────────────────┘
               │
               ├──────▶ Model 업데이트
               │        (soapLevel 감소)
               │
               └──────▶ 파티클 재생
                        (시각 효과)

Model 변경됨!
      │
      ▼
OnDataChanged 이벤트 발송! 📢
      │
      ├──────▶ View 반응 → 게이지 업데이트
      └──────▶ 기타 구독자 반응
```

## 왜 이 구조가 좋은가요?

### 1. 유지보수 용이
- UI 디자인 바꾸고 싶다? → View만 수정
- 분사 시간 바꾸고 싶다? → Controller만 수정
- 저장 형식 바꾸고 싶다? → Model만 수정

### 2. 테스트 용이
- Controller를 별도로 테스트할 수 있어요
- Mock View를 만들어서 UI 없이 로직 테스트 가능

### 3. 협업 용이
- 디자이너는 View 작업
- 프로그래머는 Controller/Model 작업
- 서로 충돌 없이 동시 작업 가능!

## 자주 묻는 질문

**Q: MVVM, MVP 같은 다른 패턴도 있던데요?**

A: 네, 다양한 패턴이 있어요. MVC는 가장 기본적이고 이해하기 쉬워서
   입문용으로 좋아요. 프로젝트 규모에 따라 다른 패턴을 선택하기도 해요.

**Q: Model과 View가 직접 통신해도 되나요?**

A: 이 프로젝트에서는 이벤트를 통해 간접적으로 통신해요.
   Model이 변경되면 이벤트를 발송하고, View가 구독해서 반응하죠.
   직접 참조보다 느슨한 결합(Loose Coupling)으로 유연성이 높아요.
